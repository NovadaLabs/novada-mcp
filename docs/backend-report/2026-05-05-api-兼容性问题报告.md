## 执行摘要（请先读这里）

### 用一句话说清楚问题

> 我们的代码没有写错。Novada Dashboard 和外部 API 调用走的是两套完全独立的权限体系，Dashboard 使用的 Cookie 认证路径自动拥有所有已订阅功能，而 Bearer Token API 路径的对应权限从未被开通过。

### 四个 API 的现状一览

| API | Dashboard | 直接 API 调用 | 错误 | 根本原因 |
|-----|-----------|--------------|------|----------|
| **Scraper API** | ✅ 正常 | ❌ 失败 | 11006 | Bearer Token 权限未开通 |
| **SERP 搜索** | ✅ 正常 | ❌ 失败 | 402 权限拒绝 | Bearer Token 权限未开通 |
| **Browser API** | ⚠️ 卡死 | ❌ 超时 | 连接挂起 | 服务端残留会话阻塞新连接 |
| **Web Unblocker** | ✅ 正常 | ✅ 基本可用 | 格式缺陷 | response_format 仅支持 html |

### 为什么 Dashboard 能用，API 不能用？

Novada 内部有两套权限系统：

```
用户账户订阅了 Scraper 功能
        │
        ├──▶ Cookie Session 权限（Dashboard 使用）  ✅ 自动继承订阅权限
        │
        └──▶ Bearer Token 权限（API 直接调用）      ❌ 独立维护，从未激活
```

用户在 Dashboard 上能看到并使用某个功能，并不代表他的 API Key 也有该功能的调用权限。这两条路径在 Novada 后端是完全分开的，互不同步。这是整份报告所有问题的共同根源。

- **结果：** `Target.getBrowserContexts timed out. Increase the 'protocolTimeout' setting...`
- **说明：** WebSocket 连接成功建立，但 CDP 协议层无响应

**测试 3（2026-05-05）：Dashboard Playground 对比**

通过用户浏览器在 `dashboard.novada.com/overview/browser/` 点击 "Run" 执行 Playground 示例：

- **结果：** 控制台显示 `Status running`，**持续超过 2 分钟仍未完成**
- **说明：** Dashboard 本身也无法完成连接，Playground 卡住

#### 关键发现（来自官方文档）

通过读取 `developer.novada.com/novada/advanced-proxy-solutions/browser-api/configure-browser-api` 发现：

> **"one active session per account"（每个账户同时只允许一个活跃会话）**

官方文档原文：
- 每个账户只能维持一个活跃的浏览器会话
- 如果上一个会话没有显式关闭（`browser.close()`），新的连接请求会**无限期挂起**
- 会话 TTL：30 分钟无操作后自动关闭

#### 根本原因分析

| 问题 | 原因 |
|------|------|
| connectOverCDP 超时 | 账户已有一个活跃会话未关闭，新连接被服务端挂起 |
| puppeteer.connect 超时 | 同上 |
| Dashboard Playground 卡住 | 同上——说明不是我们代码的问题，是服务端会话状态问题 |
| 会话是谁创建的 | 可能是之前测试过程中某次连接没有正确关闭，残留在服务端 |

**结论：连接方法（connectOverCDP）是正确的**（官方文档确认）。问题是**服务端存在一个卡死的残留会话，阻塞了所有新连接**。

### 3.4 Web Unblocker（网页解锁）

#### 接口信息
- **端点：** `POST https://webunlocker.novada.com/request`
- **认证：** `Authorization: Bearer NOVADA_WEB_UNBLOCKER_KEY`

#### 测试记录

**第一阶段（2026-04-20）：接口调通，发现格式问题**

- 测试次数：5 次
- 发现的格式问题（已修复）：
  - `js_render` 必须是布尔值 `true`，不能是字符串 `"True"`
  - 路径必须是 `/request`，不能是根路径
  - 认证是 Bearer Token，不是 body 里的 `api_key` 字段
- 修复后状态：✅ 基本功能可用

**第二阶段（2026-05-05）：发现两个后端问题**

#### 问题 A：`response_format` 参数实际只支持 `"html"`

官方文档列出了三种响应格式：`html`、`json`、`markdown`。测试结果：

| response_format 值 | 结果 |
|--------------------|------|
| `"html"` | ✅ 正常返回 |
| `"json"` | ❌ 返回错误码 `11008` |
| `"markdown"` | ❌ 返回错误码 `11008` |

我们的代码已将 `response_format` 硬编码为 `"html"`，暂时规避了此问题。但官方文档中仍有 `json` 和 `markdown` 的说明，会误导使用者。

#### 问题 B：HTTP 状态码全部返回 200，错误隐藏在 JSON body 中

无论请求成功还是失败，HTTP 状态码始终为 `200`。错误信息嵌套在响应 JSON 的内部字段中：

```json
{
  "code": 0,           ← 外层 code（始终为 0，无意义）
  "data": {
    "code": 11008,     ← 真实错误码（隐藏在这里）
    "msg": "xxx",
    "html": null
  }
}
```

这导致：如果调用方只检查 HTTP 状态码（正常做法），会认为请求成功，实际上拿到的是错误响应。我们已在代码中增加内层 `data.code` 的检查，但这属于后端设计问题。

#### 发现的其他问题

- **内部主机名泄露：** 错误响应的 `err_detail` 字段中暴露了内部服务器主机名（如 `proxy-unblock-na.novada.pro`），建议生产环境过滤

#### 所需后端操作

1. **修复 `response_format` 参数：**
   - 选项 A：实现 `json` 和 `markdown` 格式的支持
   - 选项 B：删除文档中对这两种格式的说明，明确只支持 `html`

2. **修复 HTTP 状态码：**
   - API 错误应返回对应 HTTP 错误码（4xx/5xx），而不是全部返回 200
   - 至少对以下情况使用正确状态码：401（认证失败）、403（权限不足）、400（参数错误）

3. **清理 `err_detail` 中的内部主机名**（安全建议，P2 优先级）

## 四、核心问题：为什么 Dashboard 能用，API 不能用？

这是本报告最重要的发现，需要后端团队理解并解决。

### 对比矩阵

| API 功能 | Dashboard 使用方式 | 直接 API 使用方式 | 结果 |
|----------|-------------------|-------------------|------|
| 结构化抓取 | Cookie Session → 内部端点（不对外） | Bearer Token → `scraper.novada.com/request` | Dashboard ✅ / API ❌ 11006 |
| SERP 搜索 | Cookie Session → 内部端点 | Bearer Token → `scraperapi.novada.com/search` | Dashboard ✅ / API ❌ 402 |
| 云端浏览器 | 浏览器内 WebSocket（带登录 Cookie） | Node.js WebSocket + CDP | Dashboard ⚠️ 卡死 / API ❌ 超时 |
| Web 解锁 | Cookie Session → 内部调用 | Bearer Token → `webunlocker.novada.com/request` | Dashboard ✅ / API ✅（有格式缺陷） |

### 根本原因

Dashboard 使用的是**内部认证路径（Cookie + Session）**，而外部 API 调用使用的是**Bearer Token 路径**。这两条路径在权限系统中是**独立的**：

- Cookie 路径：Dashboard 用户登录后自动拥有，不需要额外激活
- Bearer Token 路径：需要单独为每个 API Key 开通特定功能权限

**目前的状态：该 API Key 仅激活了 Bearer Token 路径中的基础功能（代理、Web Unblocker），但 Scraper 和 SERP 功能的 Bearer Token 权限未开通。**

这就是为什么 Dashboard 可以工作（Cookie 路径），而 MCP 代码调用失败（Bearer Token 路径权限不足）。

## 九、架构建议：一次性根本解决方案

### 当前问题的本质

逐个 API Key 手动激活权限不是最终解法——它只是治标，每新增一个客户或 API Key 都需要重复操作，长期会成为运营负担。

当前权限模型存在根本性的设计裂缝：

```
用户账户
├── Cookie Session 权限（Dashboard 使用）
│   └── Scraper ✅  SERP ✅  Browser ✅  ← 开通订阅后自动拥有
│
└── Bearer Token 权限（API 直接调用）
    └── Scraper ❌  SERP ❌              ← 需要额外手动激活，不自动继承
```

**结果：** 同一个用户，在 Dashboard 上能用的功能，通过 API 调用却失败。这对开发者来说是完全无法理解的体验断层，也是目前所有问题的共同根源。

---

### 推荐方案：统一权限来源（Option B）

**核心思路：** Bearer Token 的可访问功能 = 该用户账户的订阅功能。两者使用同一套权限数据源，不再维护两套独立的激活状态。

**改造逻辑：**

```
改造前（当前）：
  验证 Bearer Token
    → 查询 bearer_token_permissions 表
    → 检查是否单独激活了 scraper_access = true
    → ❌ 未激活 → 返回 11006

改造后（目标）：
  验证 Bearer Token
    → 解析出 user_id / account_id
    → 查询该账户的订阅套餐（与 Dashboard Cookie 用同一张表）
    → 检查套餐是否包含 scraper 功能
    → ✅ 包含 → 正常处理请求
```

---

### 实施步骤

| 步骤 | 操作 | 说明 |
|------|------|------|
| 1 | 审查现有权限数据结构 | 确认 Cookie Session 和 Bearer Token 当前分别查哪张表/哪个字段 |
| 2 | 确定"功能权限"的统一来源 | 通常是用户订阅套餐表（`subscriptions` 或 `account_products`） |
| 3 | 修改 Bearer Token 鉴权中间件 | 将功能检查改为查订阅套餐，而不是独立的 bearer 权限表 |
| 4 | 回归测试 | 确认 Dashboard（Cookie）和 API（Bearer）对同一用户返回一致结果 |
| 5 | 清理旧逻辑 | 删除或归档手动激活 bearer_token_permissions 的相关代码和入口 |

**预估工时：** 3–5 个工作日（含测试）  
**一次性：** ✅ 修改后对所有现有和未来用户永久生效，无需再手动激活

---

### 过渡期处理（实施期间）

在 Option B 完成之前，建议运行一次性迁移脚本作为临时措施：

```sql
-- 伪代码：将所有已在 Dashboard 激活某产品的账户，批量同步 Bearer 权限
INSERT INTO bearer_token_permissions (api_key, product, activated_at)
SELECT ak.key, sp.product_name, NOW()
FROM api_keys ak
JOIN account_products sp ON ak.account_id = sp.account_id
WHERE sp.is_active = true
ON CONFLICT DO NOTHING;
```

这个脚本运行一次，解决所有存量账户（包括 `1f35b477...`），之后 Option B 上线后即可废弃。

---

### 为什么不选 Option A（逐个激活脚本）

| 维度 | Option A（脚本逐个激活） | Option B（统一权限源） |
|------|--------------------------|------------------------|
| 实施工时 | 1 天 | 3–5 天 |
| 是否永久解决 | ❌ 新客户仍需手动操作 | ✅ 一次性 |
| 运营负担 | 持续存在 | 消除 |
| 开发者体验 | 仍有激活断层 | Dashboard 和 API 完全一致 |
| 扩展性 | 差 | 好 |

Option A 是临时补丁，Option B 是根本修复。建议 Ethan 审批 Option B 的开发排期，同时用 Option A 的过渡脚本先解锁测试账号。
