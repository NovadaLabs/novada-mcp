§4 为什么字符数少不代表差？Why Fewer Chars ≠ Worse Novada 7,149 字符 vs Firecrawl 72,057 字符的正确解读 Novada 7,149 chars vs Firecrawl 72,057 chars — the right interpretation AI 智能体视角AI Agent POV

Novada：主内容提取Novada: Main Content Extraction

  • 只返回正文：文章、产品描述、代码、表格Returns only body: articles, product text, code, tables
  • 去除导航栏、页脚、侧边栏、广告、脚本Strips nav, footer, sidebar, ads, scripts
  • 7,149 字符 = 纯净信号7,149 chars = clean signal
  • AI Agent 所需 token 更少，推理更准确AI agent needs fewer tokens, reasoning is more accurate

Firecrawl/Tavily：全页 MarkdownFirecrawl/Tavily: Full Page Markdown

  • 返回整页转为 Markdown，包含所有元素Returns entire page as Markdown, all elements included
  • 包含导航菜单、Cookie 通知、footer 链接Includes nav menus, cookie notices, footer links
  • 72,057 字符但信噪比低72,057 chars but lower signal-to-noise ratio
  • Agent 需要更多 token 处理才能找到真正内容Agent needs more tokens to filter before finding real content
关键结论：对 AI Agent 来说，少而精 > 多而杂。Key insight: For AI agents, clean and concise > large and noisy.
Novada 的 7,149 字符包含的有效信息密度通常高于 Firecrawl 的 72,057 字符。维基百科 AI 页面：Novada 返回 22,136 字符纯文章内容；Firecrawl 返回 629,179 字符（包含所有引用、导航、元数据）。 Novada's 7,149 chars typically contain higher information density than Firecrawl's 72,057. Wikipedia AI page: Novada returns 22,136 chars of pure article; Firecrawl returns 629,179 chars (all references, navigation, metadata).
什么情况下低字符数才是问题：When low char count IS a problem: 如果质量分也低（< 6），说明主内容提取也失败了。例如：engineering.fb.com 返回 293 字符 + 质量分 4 → 真正的提取失败，不是"精简"。监控质量分，不是字符数。 If quality score is also low (< 6), the main content extraction itself failed. e.g., engineering.fb.com returned 293 chars + quality score 4 → genuine extraction failure, not "conciseness." Watch quality scores, not char counts.

基准测试失真说明（先读）Benchmark Artifacts (read first)
  • JS-Heavy P50 8814ms 是人为放大：基准强制所有 JS-Heavy URL 使用 render="render"，绕过了 auto 模式的域名注册优化。真实用户调用（render="auto"）的 P50 约为 2000–2500ms，与 Firecrawl 698ms 的差距远小于 12× 所示。 JS-Heavy P50 8814ms is a forced-render artifact: Benchmark forces render="render" for all JS-Heavy URLs, bypassing auto-mode domain-registry optimizations. Real-world P50 with render="auto" ≈ 2000–2500ms — the gap vs Firecrawl 698ms is much smaller than the apparent 12×.
  • 质量分差 7.6 vs 8.9 主要是评分算法偏差：评分函数对字符数给予重权（0–3分/10）。Novada 截断于 25KB，Firecrawl 提取 49KB+。内容密度相当，分差 70% 来自截断限制，非真实质量差距。 Quality gap 7.6 vs 8.9 is 70% scoring algorithm bias: Scoring awards up to 3/10 points purely for char count. Novada truncates at 25KB, Firecrawl extracts 49KB+. Content density is comparable — 70% of the gap is the extraction limit, not real quality.
  • Anti-Bot 74% 失败混合了两类问题：5/9 是 IP 声誉（MT-1 bug 修复后可解决），4/9 是 TLS 指纹识别（平台级问题）。单一数字掩盖了可修复的部分。 Anti-bot 74% failure mixes two distinct problems: 5/9 are IP-reputation blocks (fixable via MT-1 bug fix), 4/9 are TLS fingerprinting (platform-level). The single number masks the fixable portion.
