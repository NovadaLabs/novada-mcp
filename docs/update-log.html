A 修复 Scraper 状态竞态 Fix Scraper Status Race Condition ✓ 已提交并推送✓ Committed & Pushed

src/tools/scraper_status.ts · commit e708010

问题Problem

在 novada_scraper_submit 成功返回 task_id 后立即调用 novada_scraper_status,服务器尚未完成 task 传播,返回 not_found。原响应未区分「真正无效的 task_id」与「刚提交、几秒内尚未生效」这一竞态,导致 agent 误判任务失败。 Calling novada_scraper_status immediately after novada_scraper_submit returns a task_id hits a window where the server has not yet propagated the task, returning not_found. The old response did not distinguish a genuinely invalid task_id from the just-submitted propagation race — causing the agent to wrongly conclude the task had failed.

修复Fix

not_found 响应的 agent_instruction 现在明确告知:刚提交属正常现象,等待 5–10 秒后仅重试一次;若单次重试后仍 not_found,则 task_id 很可能无效或已过期(任务 24 小时后过期)。 The not_found response's agent_instruction now states that a fresh submit is normal, advises waiting 5–10s and retrying exactly once; if it is still not_found after that single retry, the task_id is likely invalid or expired (tasks expire after 24h).

Agent-First 原则:Agent-First principle: 错误响应必须携带可执行的 agent_instruction,告诉 agent「下一步做什么」,而非仅报告失败。 Error responses must carry an actionable agent_instruction telling the agent what to do next, not merely report failure.
