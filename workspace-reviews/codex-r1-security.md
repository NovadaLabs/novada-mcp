# Security Review — Codex R1

## CRITICAL (must fix before publish)
- [src/index.ts:492] Issue: `server.onerror` logs raw error objects with `console.error("[novada]", error)`, and the process-level fatal handler also logs raw errors at `src/index.ts:720`. Axios errors can include `config.headers.Authorization`, request bodies, query params, and proxy URLs. | Risk: API keys, bearer tokens, proxy passwords, or customer target URLs can leak into stderr/MCP logs. | Fix: Route all logging through a redaction helper before logging. Log only `name`, sanitized `message`, status code, Novada error code, and non-sensitive metadata; never log raw Axios/config/request objects.
- [src/tools/proxy_residential.ts:94] Issue: `format="env"` returns `export HTTP_PROXY="${proxyUrl}"` with the real proxy password; `format="curl"` does the same at line 112. | Risk: Proxy credentials are exposed in MCP responses/transcripts and can be copied into downstream logs by agents. | Fix: Default to masked proxy URLs in all responses. If a credential-bearing output is required, gate it behind an explicit `reveal_credentials: true` boolean and warn that output contains secrets, or return structured `{ host, port, username, password_env: "NOVADA_PROXY_PASS" }` instead.
- [src/tools/proxy_static.ts:84] Issue: `format="env"` returns `export HTTP_PROXY="${proxyUrl}"` with the real proxy password; `format="curl"` does the same at line 103. | Risk: Static proxy credentials leak through MCP output and logs. | Fix: Same as residential proxy: mask by default and require an explicit reveal path, preferably using environment-variable placeholders rather than embedding `NOVADA_PROXY_PASS`.

## HIGH
- [src/tools/scraper_status.ts:86] Issue: The API key is sent as both a bearer token and query parameter (`params: { apikey: apiKey }`). | Risk: Query-string credentials are commonly captured by access logs, reverse proxies, APM tools, and error reports. This also increases blast radius if an Axios error is logged raw. | Fix: Prefer only `Authorization: Bearer ...`. If Novada requires `apikey`, isolate this in a narrow helper and ensure all logs/redaction cover `apikey` in URLs, params objects, and serialized errors.
- [src/tools/scraper_result.ts:132] Issue: The result endpoint URL is manually constructed with `&apikey=${encodeURIComponent(apiKey)}`. | Risk: The full URL contains the API key and can leak through HTTP logs, Axios config, thrown errors, or support traces. | Fix: Use bearer auth if supported. If the legacy download endpoint requires a query key, construct via `axios.get(base, { params })`, never include full URLs in errors/responses, and aggressively redact `apikey` from any serialized config.
- [src/tools/scraper_result.ts:203] Issue: Fallback status request sends `Authorization: Bearer ${apiKey}` and also `params: { apikey: apiKey }`. | Risk: Same query-string credential leakage as scraper status. | Fix: Remove duplicate query auth if possible; otherwise centralize and redact it.
- [src/_core/auth.ts:15] Issue: `getApiKey()` only trims `NOVADA_API_KEY`; it does not reject control characters, whitespace inside the key, or malformed token characters. `src/index.ts:97` bypasses `getApiKey()` entirely and stores the raw env var at module load. | Risk: Malformed secrets can flow directly into HTTP headers, causing header errors, log noise, or potential header-injection attempts in less strict runtimes/adapters. Bypassing the helper also loses consistent validation and trimming. | Fix: Add a strict token regex/length check in `getApiKey()` such as printable token chars with no CR/LF/control chars, and call `getApiKey()` at request time in `index.ts` instead of reading `process.env.NOVADA_API_KEY` directly.
- [src/tools/scraper_submit.ts:9] Issue: `url` only validates `.url()` and `http(s)` scheme; it does not block localhost, loopback, link-local, private IP ranges, IPv6 private/loopback, or internal hostnames. | Risk: User-controlled URLs are passed to Novada's scraper service and may enable SSRF against Novada/cloud infrastructure or internal metadata endpoints. | Fix: Reuse the stricter `safeUrl` validator from `src/tools/types.ts` or move it into `_core` and apply it here.
- [src/tools/browser_flow.ts:31] Issue: Browser Flow `url` has the same `.url()` plus scheme-only validation and lacks public-host enforcement. | Risk: Cloud browser navigation can be used for SSRF or internal network probing from Novada's browser environment. | Fix: Apply the shared safe URL validator that rejects localhost/private/link-local/reserved IPs and internal-only hostnames before sending the payload.

## MEDIUM
- [src/_core/errors.ts:150] Issue: `sanitizeMessage()` redacts `api_key=`, `apikey=`, and `Authorization: Bearer`, but does not redact bare `Bearer ...`, JSON fields like `"api_key":"..."`, `NOVADA_API_KEY=...`, proxy URLs with `user:pass@host`, `NOVADA_PROXY_PASS=...`, or non-scraper Novada URLs. | Risk: Secret redaction is incomplete; any future error path that includes config, env strings, or proxy URLs can leak credentials. | Fix: Expand redaction coverage and unit-test it with bearer headers, query params, JSON bodies, env assignment strings, and proxy URLs.
- [src/_core/errors.ts:178] Issue: Existing `NovadaError` instances are returned without re-sanitizing `message` or `detail`; `makeNovadaError()` also stores caller-provided messages verbatim. | Risk: A tool that passes server-provided text or an exception message into `makeNovadaError()` can bypass `sanitizeMessage()` and surface secrets. | Fix: Sanitize `opts.message` and `opts.detail` in the `NovadaError` constructor or sanitize in `makeNovadaError()` and before returning pass-through `NovadaError`s.
- [src/tools/scraper_submit.ts:153] Issue: `serverMsg = body?.msg ?? err.message` is included in a returned NovadaError without local redaction. | Risk: A remote API response or Axios message that reflects request details can leak sensitive URL/query/header content. | Fix: Sanitize server messages before building user-facing errors, or rely on a sanitized `NovadaError` constructor as above.
- [src/tools/browser_flow.ts:159] Issue: `serverMsg` from the Browser Flow API/Axios error is interpolated into returned errors at lines 164 and 170 without local redaction. | Risk: Reflected target URLs, action values, or credentials in upstream messages can be exposed to the caller/logs. | Fix: Sanitize upstream messages before returning them.
- [src/tools/proxy_residential.ts:8] Issue: `url` is declared as `z.string().optional()` with no `.url()`/safe URL validation. It is currently unused, despite the description claiming scoped config for that URL. | Risk: If later wired into username construction or API payloads, this becomes an input-validation gap; current schema also misleads callers into thinking target URLs are validated/scoped. | Fix: Remove the unused param or validate it with the shared safe URL validator before any future use.
- [src/tools/proxy_static.ts:8] Issue: `url` is also an unused, unconstrained `z.string().optional()`. | Risk: Same latent validation gap as residential proxy. | Fix: Remove or validate with the shared safe URL validator.
- [src/tools/browser_flow.ts:11] Issue: Browser action `selector` and `value` are unconstrained strings with no length limits or character policy. They flow directly into the Browser Flow API payload. | Risk: Large values or malformed selectors can cause excessive remote work, noisy errors, or injection into downstream browser automation semantics. | Fix: Add `.max()` limits to both fields, require selector for click/type actions, and constrain `scroll` values to an enum (`up`/`down`) instead of arbitrary text.
- [src/index.ts:542] Issue: The disabled-tool error reflects raw `NOVADA_GROUPS` back to the caller. | Risk: Low-grade configuration leakage and response/log injection if the env var contains unexpected control characters. | Fix: Validate `NOVADA_GROUPS` with a regex during startup and print a sanitized/normalized value only.
