# novada-mcp 0.7.8 — Benchmark-Driven Improvements
> Session: 2026-05-22 | Source: workspace/benchmark/report.html (30-round live run)

## What changed

### P0-1 — Chainable Output on all success responses
**Files:** `src/tools/scrape.ts`, `src/tools/search.ts`, `src/tools/crawl.ts`

Every success response now appends a `## Chainable Output` section with:
- `source_url` / `root_url` / `top_urls` — structured URLs the agent can act on directly
- `agent_instruction` — explicit next-step guidance on every success (not just errors)

**Why:** AF score measurement showed `output_is_chainable = 0%` and `has_agent_instruction = 0%` on success across all 30 rounds. Firecrawl returns `sourceURL + links[]` in metadata → 100% chainable (AF 4/5 vs Novada's 3/5). Fix expected to move Novada to 5/5 AF score.

---

### P1-1 — Parallel JS re-fetch in crawl
**File:** `src/tools/crawl.ts:127-138`

JS-heavy page re-fetching inside crawl changed from sequential `for...await` to `Promise.all`.

**Before:**
```ts
for (let i = 0; i < pages.length; i++) {
  if (pages[i] !== null && detectJsHeavyContent(pages[i]!.html)) {
    pages[i] = await fetchPage(batch[i].url, apiKey, true); // sequential
  }
}
```

**After:** `Promise.all(jsHeavyIndexes.map(i => fetchPage(...)))` — all JS-heavy pages in a batch re-fetched simultaneously.

**Expected impact:** T5 crawl latency 25s → ~15s (-40%). Scales with JS-heavy pages per batch (up to CRAWL_CONCURRENCY = 3).

---

### P1-2 — Static fast path in extract
**File:** `src/tools/extract.ts:165-175`

For `render="auto"` mode, now races a direct HTTP fetch (no proxy, 3s timeout) against `fetchViaProxy` simultaneously. Direct fetch "wins" only if it returns clean HTML (no bot challenge, no JS-heavy indicators). Bot-protected or JS-heavy pages: direct rejects, proxy result used — no behavior change.

**Expected impact:** Open static sites (HN, Wikipedia, TechCrunch) drop from ~3.5s (proxy) to ~300ms (direct). ~40-60% of typical agent web access patterns. Bot-protected sites unaffected.

---

### P2-1 — `enrich_top` param on novada_search
**Files:** `src/tools/types.ts`, `src/tools/search.ts`

Added `enrich_top: boolean (optional, default false)` to search params. When `true` and no `extract_options` is set, auto-extracts the top result's full content inline. Shorthand for `extract_options: { top_n: 1 }`.

**Expected impact:** Search quality Q4 → Q5 for queries where top result has all needed content.
