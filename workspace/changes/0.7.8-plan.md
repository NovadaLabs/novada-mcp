## Phase 1 — Response trust signals
> Status: DONE (2026-05-22)
> Estimated effort: ~1 hour
> Files: `src/tools/extract.ts`, `src/tools/search.ts`, `src/tools/crawl.ts`, `src/tools/scrape.ts`

### B — `source` field on every response

Every response that returns fetched content adds:
```
source: live          # live HTTP fetch (proxy or direct)
source: wayback       # fell back to Wayback Machine archive
source: cache         # served from session dedup cache (Phase 3)
fetched_at: 2026-05-22T10:23:11Z   # already present in extract, add to search/crawl
```

**Why:** Agents can't currently distinguish live content from archived content. If `novada_extract` returns a product pricing page via wayback, the agent may trust a stale price. The `source` field closes that loop.

**Implementation:**
- `extract.ts`: wayback fallback already sets a flag (`usedWayback`?). Expose it as `source: "wayback"` in the response header line.
- `search.ts`: always `source: live` (SERP results are always fresh)
- `crawl.ts`: aggregate — `source: live (N pages) | wayback (M pages)` if any wayback used
- `scrape.ts`: always `source: live` (Scraper API is always real-time)

**Locate wayback fallback in extract.ts:**
```bash
grep -n "wayback\|archive.org\|usedWayback" src/tools/extract.ts src/utils/http.ts
```

### C — Failure classification: transient vs permanent

Current error responses say `"do not retry automatically"` as a blanket rule. This is too conservative. Add to every error `agent_instruction`:

```
retry_recommended: true | false
retry_after_ms: 5000        # only when retry_recommended: true
failure_class: transient | permanent | auth | quota
```

| HTTP status / error | Class | Retry |
|---------------------|-------|-------|
| 429 rate limit | quota | true, after 5000ms |
| 503, 502, timeout | transient | true, after 3000ms |
| 404 not found | permanent | false |
| 400 bad params | permanent | false |
| 401, 403 auth | auth | false |
| bot challenge | transient | true, try render="render" |

**Files to update:** Error handler in `src/_core/errors.ts` + `src/tools/index.ts` (classifyError function).

### Verify Phase 1
- Extract a known 404 URL → response includes `failure_class: permanent, retry_recommended: false`
- Extract a live URL → response header includes `source: live`
- If wayback fallback triggers → response shows `source: wayback`

## Phase 2 — Output format + agent memory
> Status: DONE (2026-05-22)
> Estimated effort: ~3 hours
> Files: `src/tools/search.ts`, `src/tools/crawl.ts`, `src/tools/types.ts`, all tools

### A — JSON mode for novada_search + novada_crawl

`novada_extract` already has `format: "json"`. Add it to search and crawl.

**Search JSON output:**
```json
{
  "status": "ok",
  "query": "...",
  "engine": "google",
  "result_count": 8,
  "results": [
    { "rank": 1, "title": "...", "url": "...", "snippet": "...", "published": "..." },
    ...
  ],
  "agent_instruction": "Search complete. Call novada_extract with results[0].url to read full page."
}
```

**Crawl JSON output:**
```json
{
  "status": "ok",
  "root_url": "...",
  "pages_crawled": 5,
  "pages": [
    { "url": "...", "title": "...", "depth": 1, "word_count": 423, "text": "..." },
    ...
  ],
  "agent_instruction": "..."
}
```

**Types changes:**
- `SearchParams`: add `format: z.enum(["markdown", "json"]).default("markdown")`
- `CrawlParams`: same

**Why:** Agents doing multi-hop research currently parse markdown strings to extract URLs. JSON lets them do `results[0].url` directly. This is the clearest "agent-first" signal. Firecrawl built their entire API around structured JSON output.

### D — `remember:` hint field

After each successful extract/search/crawl, append a `remember:` field with the single most valuable storable fact from that response.

```
## Agent Memory
remember: "iPhone 17 Pro Max price on Apple.com: $1,199 as of 2026-05-22"
remember: "Python 3.13 docs at https://docs.python.org/3.13/ — 847 pages crawled"
```

**Implementation approach:** Simple heuristic extraction:
- For product pages: look for price patterns (`$X,XXX`)
- For documentation crawls: record root URL + page count
- For search: record query + top result URL
- For news: record headline + publication date

This is unique positioning: Novada is the only scraping MCP that thinks about agent memory output, not just agent input. Compatible with AgentRecall `remember()` API.

## Phase 3 — Session dedup cache
> Status: DONE (2026-05-22)
> Estimated effort: ~2 hours
> Files: `src/tools/extract.ts`, new `src/_core/session-cache.ts`

### E — In-process session cache for extract

In research loops, agents frequently call `novada_extract` on the same URL multiple times (different sub-agents, no shared memory). Each call hits the API and costs quota.

**Design:**
```ts
// src/_core/session-cache.ts
const cache = new Map<string, { result: string; ts: number }>();
const TTL_MS = 5 * 60 * 1000; // 5 minutes

export function getCached(url: string, renderMode: string): string | null
export function setCached(url: string, renderMode: string, result: string): void
```

Cache key: `${url}::${renderMode}` (e.g., `https://example.com::auto`)

**Integration in `extractSingle`:** Check cache before any fetch. On cache hit, return immediately with `source: cache` added to response header. On cache miss, proceed normally and store result.

**Scope:** Only caches `extractSingle` (single-URL extract). Not applied to batch extract, search, crawl — those are inherently time-sensitive or multi-result.

**Why in-process (not Redis):** Agents are session-scoped. Cross-session caching would serve stale content. A process-local Map is zero-dependency, zero-latency, and discards on restart — exactly the right scope.
