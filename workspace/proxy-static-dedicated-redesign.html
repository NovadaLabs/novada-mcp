Proxy Static & Dedicated: Architectural Redesign

Engineering proposal to fix fundamentally wrong proxy credential generation for static and dedicated IP products.

1Executive Summary

Critical Bug — 100% Failure Rate Both novada_proxy_static and novada_proxy_dedicated tools generate zone-based proxy URLs (e.g. user-zone-static-country-us-session-demo:pass@endpoint:7777) for products that use a completely different authentication model. Every command produced by these tools will fail when executed against a real static or dedicated IP.

Static and dedicated proxies are IP-based products: each IP address is purchased individually from the Novada dashboard and assigned its own fixed IP, unique port, and per-IP username/password credentials. These IPs do not route through the shared zone gateway and do not accept zone-format usernames.

Dimension Zone-Based (Residential/ISP) IP-Based (Static/Dedicated)
Endpoint Shared gateway (e.g. gate.novada.com:7777) Unique IP per proxy (e.g. 151.242.47.74:8886)
Username Zone format: user-zone-static-country-us-session-X Per-IP credential: ax0kSJ8snE6wF1mR
Password Account master password Per-IP random password (e.g. p3K0rNpsP2iR)
Provisioning Automatic (zone routing) Manual purchase at dashboard.novada.com
curl format curl --proxy "http://user:pass@gate:7777" URL curl -x 151.242.47.74:8886 -U "user:pass" URL

Impact: Any agent or user who calls novada_proxy_static or novada_proxy_dedicated receives a completely invalid proxy configuration. The correct fix is a full architectural redesign of both tools.

2Current Architecture (What's Wrong)

Current Code Pattern (proxy_static.ts)

Both tools share the same flawed pattern — they call getProxyCredentials() which reads NOVADA_PROXY_USER, NOVADA_PROXY_PASS, and NOVADA_PROXY_ENDPOINT, then construct a zone-format username:

proxy_static.ts (current — WRONG)// Current username builder — WRONG for static/dedicated
function buildStaticUsername(user: string, params: ProxyStaticParams): string {
  const parts = [user];
  parts.push("zone-static");
  parts.push(`country-${params.country.toLowerCase()}`);
  parts.push(`session-${params.session_id}`);
  return parts.join("-");
}
// Produces: "user-zone-static-country-us-session-demo"
// Then routes to: NOVADA_PROXY_ENDPOINT (shared zone gateway)
// ⚠ This is NEVER the correct format for static/dedicated IPs

Why This Is Wrong

Root Cause The zone-based credential model (shared gateway + zone routing) only applies to Residential, ISP, and Mobile proxy products. Static and Dedicated IPs are provisioned differently — they are individual IPs purchased outright, each with unique fixed credentials. Using zone routing against these IPs will always result in authentication failure.
Produced by Current Code What Actually Works
curl --proxy "http://user-zone-static-country-us-session-abc:pass@gate.novada.com:7777" https://ipinfo.io curl -x 151.242.47.74:8886 -U "ax0kSJ8snE6wF1mR:p3K0rNpsP2iR" https://ipinfo.io
Uses shared gateway Uses unique IP:PORT per proxy
Zone-format username (incorrect) Per-IP random username
Account-level password Per-IP unique password
Result: 407 / auth failure Result: success + static IP in response

3Proposed New Architecture

Core Concept: IP List Environment Variables

Since static and dedicated IPs are purchased individually and have unique per-IP credentials, the cleanest approach is to let users store their purchased IPs in structured environment variables. The MCP tools then read, parse, and serve the correct IP + credentials on demand.

New Environment Variables NOVADA_STATIC_PROXY_LIST — one static ISP IP per line
NOVADA_DEDICATED_PROXY_LIST — one dedicated datacenter IP per line

Format per line: IP:PORT:USERNAME:PASSWORD
Example: 151.242.47.74:8886:ax0kSJ8snE6wF1mR:p3K0rNpsP2iR

Static Proxy (NOVADA_STATIC_PROXY_LIST)

Static ISP proxies are organized by country — each purchased IP is associated with a specific country. The list format supports an optional 5th field for the country code: 

# NOVADA_STATIC_PROXY_LIST format (one per line)
# IP:PORT:USER:PASS[:COUNTRY_CODE]
151.242.47.74:8886:ax0kSJ8snE6wF1mR:p3K0rNpsP2iR:us
185.220.101.42:8887:bY9nKT3pmR1vW4xJ:q7L2sNqP3jS6:gb
93.188.167.55:9100:cZ0mLU4qnS2yX5kK:r8M3tOrQ4kT7:de

When the tool is called with a country parameter, it filters the list to matching entries and returns one. When no country filter is applied or no match found, it falls back to the first entry in the list.

Dedicated Proxy (NOVADA_DEDICATED_PROXY_LIST)

Dedicated datacenter proxies do not have country as a primary filter — they are exclusive IPs. The tool can return a specific proxy by index, or return all entries for the agent to choose from: 

# NOVADA_DEDICATED_PROXY_LIST format (one per line)
# IP:PORT:USER:PASS
45.152.66.201:10801:dA1oMV5rnT3zY6lL:s9N4uPsR5lU8
185.241.208.72:10802:eB2pNW6soU4aZ7mM:t0O5vQtS6mV9

Tool Behavior (New)

ScenarioTool Response
List env var not set Return configuration_required JSON with setup instructions
List set, country match found Return proxy config for matching IP in requested format
List set, no country match Return first available entry, note country mismatch
List set, multiple entries Round-robin or index-based selection
Invalid line format in list Skip malformed lines, log warning in response

4API Design

proxy_static — Updated Parameters

ParameterTypeDescription
country string (optional) ISO 2-letter country code. Filters NOVADA_STATIC_PROXY_LIST to matching entries. If omitted, returns first available entry.
session_id string (optional) Removed as required field — static IPs are always the same IP regardless of session. Retained as optional metadata only.
format enum: url | env | curl Output format. url: proxy URL string. env: shell export commands. curl: curl -x + -U flags (NOT --proxy).

proxy_dedicated — Updated Parameters

ParameterTypeDescription
index number (optional, default: 0) Which entry from NOVADA_DEDICATED_PROXY_LIST to use (0-indexed). Defaults to first entry.
format enum: url | env | curl Output format. curl format uses curl -x IP:PORT -U "USER:PASS" syntax (not --proxy).
list_all boolean (optional, default: false) If true, return all dedicated IPs in the list (useful for agents that need to assign different IPs to parallel workflows).

Output Format: curl

Key Change — curl Format The current curl format uses --proxy "http://user:pass@host:port" (zone-gateway style).
The new curl format MUST use -x IP:PORT -U "USER:PASS" which is the correct syntax for direct IP authentication. 
# OLD (wrong — zone format)
curl --proxy "http://user-zone-static-country-us-session-abc:pass@gate.novada.com:7777" https://ipinfo.io

# NEW (correct — per-IP credentials)
curl -x 151.242.47.74:8886 -U "ax0kSJ8snE6wF1mR:p3K0rNpsP2iR" https://ipinfo.io

5Migration Plan

  • 1
    Phase 1 — Early Return Guard NOW (v0.8.x)

    Add a check at the top of each tool handler: if the relevant env var (NOVADA_STATIC_PROXY_LIST / NOVADA_DEDICATED_PROXY_LIST) is not set, return a structured configuration_required response with setup instructions and dashboard link. The existing zone-based code remains but is unreachable without the env var. This prevents silent failure and gives users/agents clear instructions.
    Status: DONE (implemented in this commit)

  • 2
    Phase 2 — Full Redesign v0.9

    Parse the env var list. Replace zone-based username/endpoint logic entirely with per-IP credential extraction. Add country-based filtering for static. Add index-based selection for dedicated. Update format=curl output to use -x IP:PORT -U "USER:PASS" syntax. Update tool descriptions and Zod schemas (remove required session_id from static, add index and list_all to dedicated). Write unit tests for list parsing edge cases.

  • 3
    Phase 3 — Dashboard API Integration v1.0

    Auto-fetch the user's purchased static/dedicated IP list directly from the Novada dashboard API using NOVADA_API_KEY. Remove the need for manual env var population. The tool calls the Novada REST API, retrieves active IP inventory, and uses those credentials on the fly. Fallback to env var list if API unavailable.

6Code Examples

proxy_static.ts — Phase 2 Implementation (Before / After)

BEFORE (current — generates wrong zone format)function buildStaticUsername(user: string, params: ProxyStaticParams): string {
  const parts = [user];
  parts.push("zone-static");
  parts.push(`country-${params.country.toLowerCase()}`);
  parts.push(`session-${params.session_id}`);
  return parts.join("-");
}
AFTER (Phase 2 — reads per-IP credentials from list)interface StaticProxyEntry {
  ip: string;
  port: number;
  user: string;
  pass: string;
  country?: string;
}

function parseStaticProxyList(raw: string): StaticProxyEntry[] {
  return raw.split("\n")
    .map(line => line.trim())
    .filter(line => line && !line.startsWith("#"))
    .flatMap(line => {
      const parts = line.split(":");
      if (parts.length < 4) return []; // skip malformed lines
      const [ip, portStr, user, pass, country] = parts;
      return [{ ip, port: parseInt(portStr), user, pass, country: country?.toLowerCase() }];
    });
}

function selectStaticProxy(entries: StaticProxyEntry[], country?: string): StaticProxyEntry | null {
  if (!entries.length) return null;
  if (country) {
    const match = entries.find(e => e.country === country.toLowerCase());
    if (match) return match;
  }
  return entries[0]; // fallback to first
}

// In the handler:
const rawList = process.env.NOVADA_STATIC_PROXY_LIST!;
const entries = parseStaticProxyList(rawList);
const proxy = selectStaticProxy(entries, params.country);
if (!proxy) return JSON.stringify({ status: "error", message: "No valid entries in NOVADA_STATIC_PROXY_LIST" }, null, 2);

// curl format (correct):
const curlCmd = `curl -x ${proxy.ip}:${proxy.port} -U "${proxy.user}:${proxy.pass}" <your-url>`;

proxy_dedicated.ts — Phase 2 Implementation

AFTER (Phase 2 — dedicated)interface DedicatedProxyEntry {
  ip: string;
  port: number;
  user: string;
  pass: string;
}

function parseDedicatedProxyList(raw: string): DedicatedProxyEntry[] {
  return raw.split("\n")
    .map(line => line.trim())
    .filter(line => line && !line.startsWith("#"))
    .flatMap(line => {
      const [ip, portStr, user, pass] = line.split(":");
      if (!ip || !portStr || !user || !pass) return [];
      return [{ ip, port: parseInt(portStr), user, pass }];
    });
}

// In the handler:
const rawList = process.env.NOVADA_DEDICATED_PROXY_LIST!;
const entries = parseDedicatedProxyList(rawList);
const idx = params.index ?? 0;
const proxy = entries[idx] ?? entries[0];

// if list_all is true:
if (params.list_all) {
  return JSON.stringify({
    status: "ok",
    entries: entries.map((e, i) => ({ index: i, ip: e.ip, port: e.port, user: e.user })),
  }, null, 2);
}
