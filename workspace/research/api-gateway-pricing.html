Part 2 — AI API 中转站定价模式（国内市场）
什么是中转站：AI API 中转站（API Relay）是批量采购官方 AI API 额度（OpenAI、Anthropic、Google 等），再转售给国内用户的服务商。核心价值：解决支付问题（支持支付宝/微信）+ 解决网络问题（国内直连）+ 降低成本（批发价格）。

🔢 Token 计费结构详解

计费单位：Token

行业统一标准
  • 英文：约 1 词 = 1 Token
  • 中文：约 1–2 汉字 = 1 Token
  • 输入（Prompt）和输出（Completion）分别计费
  • 输出价格通常是输入的 3–5 倍
  • 图片/语音等多模态按特定规则折算

倍率（Rate Multiplier）

中转站的核心定价机制
  • 倍率 = 中转站价格 / 官方价格
  • 倍率 0.8 = 八折，0.3 = 三折
  • 官转合理范围：0.7–1.5
  • 倍率 < 0.5 通常意味着逆向流量掺入
  • 按「输入倍率」和「输出倍率」分别标注

充值模式

余额预付制（主流）
  • 先充值余额（支付宝/微信/银行卡）
  • 每次调用按 Token 消耗余额
  • 控制台可查看实时账单和用量
  • 部分平台支持按月订阅套餐
  • 302.AI：自定义 key 每天 $0.05 基础费

Part 3 — 数据结构 & API 协议
行业标准：中转站生态已100% 统一到 OpenAI-compatible REST API 格式。切换平台只需更换 Base URL 和 API Key，代码无需改动。

REST / JSON（主流）

所有平台支持
  • OpenAI Chat Completions 格式：/v1/chat/completions
  • 请求：{"model":"gpt-4o","messages":[...]}
  • 响应：JSON 含 choices[].message.content
  • HTTP POST，Bearer Token 认证
  • 支持同步和流式（stream:true）

SSE 流式输出

Server-Sent Events — 打字机效果
  • 设置 stream: true 开启
  • 服务端推送 data: {...} 事件流
  • 每个 chunk 含部分 delta.content
  • 结束标志：data: [DONE]
  • 逆向渠道可能存在卡顿/丢字

GraphQL

企业 API Gateway 支持
  • 支持平台：Kong、Apigee、Azure APIM、Gravitee
  • 单端点多查询，减少 over-fetching
  • Schema 验证，类型安全
  • 适合：复杂数据关系、前端驱动 API
  • 中转站生态几乎不用（专为 AI 调用设计）

gRPC

高性能二进制协议
  • 支持平台：Kong、Apigee、Gravitee、APISIX、Nginx
  • Protocol Buffers 序列化，性能是 REST 的 5-10x
  • 双向流式通信，适合微服务内部
  • 需要 HTTP/2，客户端支持复杂度高
  • AI API 中转站不支持（场景不匹配）

底层技术栈（中转站）

One API 生态
  • One API（songquanpeng/one-api）：绝大多数中转站的底层
  • 功能：多模型统一封装、Key 管理、额度控制、负载均衡、倍率设置
  • New API：社区二次开发版，更多模型支持
  • Sub2API：订阅转 API 工具（拼车用）
  • Docker 一键部署，端口 3000

支付 & 计费数据结构

账单系统常见字段
  • prompt_tokens：输入 Token 数
  • completion_tokens：输出 Token 数
  • total_tokens：合计
  • usage.cost：本次调用费用
  • Apigee / Kong 额外记录：请求延迟、来源 IP、API Plan
Key Takeaways — 核心洞察
1. 计费维度差异：API Gateway（传统）= 按请求次数计费。AI 中转站 = 按Token（输入+输出）计费。两者维度不同，不可直接比较。
2. 格式标准化：中转站生态已 100% 基于 OpenAI-compatible REST API，底层 99% 使用 One API 开源系统。切换成本极低。
3. 定价策略分化：中转站分为三类——折扣派（poloapi，3-5折，价格透明）、动态派（OpenRouter，实时路由最低价）、功能派（302.AI，按 key/天收费+功能扩展）。
4. 传统 Gateway 趋势：Kong、AWS 等开始推出 AI Gateway 模块，支持 Token 配额管理、AI 成本追踪、模型路由——传统 API Gateway 正在向 AI Gateway 演进。
Agent 使用体验 — Claude Sonnet 4.6 实测反馈
背景：以下是 Claude（我）在完成本次研究任务时，作为 AI agent 直接调用 novada-mcp v0.7.12 工具的真实体验记录。工具调用链：novada_research × 2 → novada_extract × 2批，共 9 个 URL，3 次 failed。

✅ 做得好的

对 agent 友好的设计
  • agent_instruction 字段真的有用。novada_search 返回"SERP not activated"时，错误里直接给出了替代路径（改用 novada_extract），我没有卡住，立刻调整继续。这是最重要的 agent-first 设计。
  • Batch extract 一次传 5 个 URL 并行。省去了 5 次来回，显著减少了 tool calls 数量。
  • research 末尾的 agent hints 非常清晰。"Use novada_extract with these specific URLs: …" 让我的下一步 action 是明确的，不需要猜。
  • 质量分数（75/100、45/100）帮我判断可信度。抓到 45 分的页面时我知道要多留心，没有把它当主力来源。
  • 失败也是结构化的。FAILED 的 URL 有明确标注，不会让我以为抓到了空内容。

⚠️ 有摩擦的

影响 agent 判断的地方
  • Zhihu 失败时错误太模糊。"All promises were rejected" 让我不知道是反爬、网络超时还是 render mode 选错了。如果能给出 failure_class: anti_bot 或 suggested_fix: render="render"，我可以立刻重试。
  • AWS pricing 页 403，没有渠道建议。工具知道 amazon.com 应该用哪种模式（domains.ts 里有注册），但没有把这个知识转化成"建议你用 novada_unblock"的 agent_instruction。
  • research 有 2/6 条 query 失败，但没说哪 2 条。我不知道是中文 query 失败了还是英文的，也不知道原因，无法针对性补救。
  • 大结果（83KB）自动存成文件，agent 要额外一步读取。需要调用 Bash 才能看完整内容，对纯 MCP 环境（没有文件系统访问）的 agent 会是死路。
  • render=auto 对 Zhihu 没有自动升级到 render 模式。auto 应该检测到失败后重试 render，但好像直接报错了。

💡 来自 agent 视角的改进建议

P0 — 失败时给出 suggested_fix：当 extract 失败时，在 agent_instruction 里加一行 suggested_fix: "retry with render='render' for this domain" 或 suggested_fix: "use novada_unblock for anti-bot pages"。这一个字段能让 agent 立刻知道下一步，而不是卡住。
P1 — research 失败 query 要透明：现在 queries: 4/6 succeeded 但不说哪 2 条失败了、为什么。agent 无法补救。建议加 failed_queries: ["query text", ...] 字段。
P1 — 大结果不要存文件，要截断 + 摘要：83KB 存到外部文件对没有文件系统访问的 agent（Claude Desktop、Cursor 等）是死路。建议截断到 25K chars + 末尾加摘要块，让 agent 在 tool result 里就能理解全貌。
