# Test A: Proxy Unified Key Feasibility

**Agent:** Test Agent A
**Date:** 2026-06-22
**Question:** Can NOVADA_API_KEY be used directly as proxy credentials without a separate proxy sub-account?

---

## 1. Findings from credentials.ts

File: `/Users/tongwu/Projects/novada-mcp/src/utils/credentials.ts`

`getProxyCredentials()` reads three separate env vars: `NOVADA_PROXY_USER`, `NOVADA_PROXY_PASS`, `NOVADA_PROXY_ENDPOINT`. There is **no API-key-to-proxy derivation** in this function.

The file does contain an auto-fetch path (`resolveProxyCredentials`) that:
1. Checks for explicit `NOVADA_PROXY_USER/PASS/ENDPOINT` first
2. Falls back to `NOVADA_USERNAME + NOVADA_API_KEY` → calls `POST /v1/oauth2/token` (Basic auth: `username:apiKey`) → gets an `access_token` → calls `POST /v1/proxy_account/list` to retrieve the sub-account `account` + `password`

This means credentials.ts already knows that proxy credentials are **separate** from the API key and must be fetched via the management API with a `NOVADA_USERNAME`.

## 3. Live Test Results

### Test 1: Confirm API key works as Bearer token on management API
```
POST https://api-m.novada.com/v1/wallet/balance
Authorization: Bearer 1f35b477c9e1802778ec64aee2a6adfa

Response: {"code":0,"data":{"balance":174.1},"msg":"success"}
```
**API key is valid and works for REST endpoints.**

### Test 2: API key works on proxy_account/list (no NOVADA_USERNAME needed)
```
POST https://api-m.novada.com/v1/proxy_account/list
Authorization: Bearer 1f35b477c9e1802778ec64aee2a6adfa
product=1

Response: code=0, account=tongwu_TRDI7X, password=_Asd1644asd_
```
**Key finding:** `proxy_account/list` accepts the API key as Bearer directly — no `/oauth2/token` step needed. The `NOVADA_USERNAME` requirement in `fetchProxyCredentials()` is unnecessary overhead. The API key alone is sufficient to retrieve sub-account credentials.

### Test 3: API key as proxy password — REJECTED
```
curl -x "http://tongwu_TRDI7X:1f35b477c9e1802778ec64aee2a6adfa@1b9b0a2b9011e022.vtv.na.novada.pro:7777" https://httpbin.org/ip

Response: HTTP/1.1 407 Proxy Authentication Required (AUTH_FAILED)
X-Proxy-Error-Code: AUTH_FAILED
X-Proxy-Error-Reason: The username or password is incorrect, or the account has been disabled.
```
**The API key cannot be used as the proxy password. The proxy enforces its own credential system.**

### Test 4: oauth2/token endpoint requires username (not just API key)
```
POST https://api-m.novada.com/v1/oauth2/token
Authorization: Basic base64(":apiKey")  → code 400 "authorization failure:10005"
Authorization: Basic base64("apiKey:apiKey") → code 400 "authorization failure:10003"
```
The `/oauth2/token` path in `fetchProxyCredentials()` fails without a real `NOVADA_USERNAME`. However, this path is now proven unnecessary — the API key works directly as a Bearer token on `/proxy_account/list`.

### Test 5: Proxy endpoint is live, sub-account password works (balance depleted)
Direct connection to `1b9b0a2b9011e022.vtv.na.novada.pro:7777` succeeds. The 503 responses were due to `residential_balance=0` (plan expired 2026-05-20), not authentication errors.

## 4. Architecture Conclusion

The Novada proxy system has two separate auth layers:

| Layer | Auth mechanism | Valid credentials |
|-------|---------------|-------------------|
| Management API (REST) | Bearer token | `NOVADA_API_KEY` |
| Proxy tunnel (HTTP CONNECT) | Basic auth | Sub-account `user:password` |

These are **architecturally separate**. The proxy layer does not accept the API key as a password. There is no backend shortcut.

---

## 5. Recommendation

**NO** — implementing Option A (API key as proxy password) is not feasible without backend changes. The proxy server rejects the API key with a 407.

**However**, the existing `fetchProxyCredentials()` implementation has a hidden simplification available:

- The `/oauth2/token` step is unnecessary. `NOVADA_API_KEY` alone works as a Bearer token on `/proxy_account/list` directly.
- `NOVADA_USERNAME` is **not required** — remove it from the credential resolution path.
- The simplified auto-fetch flow: `NOVADA_API_KEY` → `Bearer POST /proxy_account/list` → get `account + password` → use as proxy credentials.

**Minimum required env vars for zero-config proxy setup:**
```
NOVADA_API_KEY=<key>           # already required for all tools
NOVADA_PROXY_ENDPOINT=<host:port>  # still required (endpoint is product-specific)
```
No `NOVADA_USERNAME`, no `NOVADA_PROXY_USER`, no `NOVADA_PROXY_PASS` needed.

This reduces the setup burden from 5 env vars to 2 (or 1 if endpoint is hardcoded per product).
